# menu_relocfile_id0_collision (2026-05-10)

**Status:** RESOLVED

## Symptom

Menus (Main, 1P, VS, VS Options, Options, Data, Message — anything that
draws sprites out of the `MNCommon` library) render with heavy graphical
glitching. Concurrently the runtime log fills with stale-token errors
emitted from the sprite-library prep functions:

```
[error] RelocPointerTable: invalid/stale token 0x00D20850 (generation=0x085, max_index=242, caller=.../decomp/src/lb/lbcommon.c:3059)
[error] RelocPointerTable: invalid/stale token 0x00C90850 (generation=0x085, max_index=242, caller=.../decomp/src/lb/lbcommon.c:2816)
[error] RelocPointerTable: invalid/stale token 0x00C90850 (generation=0x085, max_index=242, caller=.../decomp/src/lb/lbcommon.c:3013)
```

Decoding the offending tokens against the live table layout (`port/resource/RelocPointerTable.cpp`,
`[12-bit generation][20-bit index]`) is the smoking gun:

| Logged token | Top 12 bits (=gen) | Low 20 bits (=index) | Inverse u16-swap (`(w<<16)|(w>>16)`) | Recovered gen |
|---|---|---|---|---|
| `0x00D20850` | `0x00D` | `0x20850` (133 200) | `0x08500D20` | **`0x085` ✓ matches live scene** |
| `0x00C90850` | `0x00C` | `0x90850` (591 952) | `0x08500C90` | **`0x085` ✓ matches live scene** |

Inverting one specific bitwise transformation lands every token's generation
field on the live scene generation exactly. That can only happen if the
tokens were originally registered correctly and then the bytes holding them
were halfword-swapped after the fact.

## Root cause

`tools/generate_yamls.py:parse_reloc_data_header` walks
`include/reloc_data.h` line-by-line and writes every `ll*FileID` define it
sees into `id_to_name[file_id]`, with later entries overwriting earlier
ones. The header is generated by `tools/generate_reloc_stubs.py`, which:

1. Emits the real values for every symbol in
   `tools/reloc_data_symbols.us.txt` first (e.g.
   `#define llMNCommonFileID ((intptr_t)0)` at line 29 — `MNCommon` legitimately
   owns file_id 0 per the vendored symbols table).
2. Appends a placeholder block of ~1770 `#define ll<Name>FileID ((intptr_t)0) /* STUBBED */`
   entries for `ll*` names that decomp src/ references but that the symbols
   table doesn't contain — they all collapse to value `0`.

Because the second block is parsed after the first, the LAST stubbed
`ll*FileID` line in the header (alphabetically `llFTYoshiAnimWin2FileID`,
since the stub list is sorted) overwrites `id_to_name[0]`. The generated
`yamls/us/reloc_animations.yml` therefore lists `FTYoshiAnimWin2:` at
file_id 0 with `MNCommon`'s actual ROM offset/size, and `MNCommon` is
absent from `yamls/us/reloc_menus.yml` entirely.

Downstream:

- `tools/generate_reloc_table.py` consumes the YAMLs and emits
  `port/resource/RelocFileTable.cpp[0] = "reloc_animations/FTYoshiAnimWin2"`.
- Torch extracts the file under that name, into the `reloc_animations/`
  directory of the `.o2r` archive.
- At load time `port/bridge/lbreloc_bridge.cpp:portRelocPathNeedsFigatreeFixup`
  matches any path starting with `reloc_animations/FT…` and runs the
  fighter-only u16 halfword-swap fixup over the file bytes.
- That fixup is correct for figatree/main-motion files, where the cells the
  reloc table marks as non-pointer are big-endian keyframe u32s that need
  the swap. Run on `MNCommon` (a sprite library), the same swap mangles
  the pointer-token cells embedded in `Sprite::LUT`, `Sprite::bitmap`,
  `Bitmap::buf`, etc.
- Every `lbCommonPrepSObjAttr` / `lbCommonPrepSObjDraw` walk then calls
  `PORT_RESOLVE` on a halfword-swapped token, the table correctly rejects
  the bogus generation field, and the renderer pumps NULL pointers into
  `gDPLoadTLUT` / `gDPSetTextureImage` — producing the visual glitches.

The error sites in the log (lines `2816`, `3013`, `3059` of
`decomp/src/lb/lbcommon.c`) are exactly the `PORT_RESOLVE(sprite->LUT)`
inside `lbCommonPrepSObjAttr` and the `PORT_RESOLVE(sprite->bitmap)` /
`PORT_RESOLVE(bitmap->buf)` inside `lbCommonPrepSObjDraw` and the
`lbCommonDrawSObj*` wrappers. The HUD, CSS glyph banks, and 3D path
draw out of *different* files (none of which match `reloc_animations/FT…`),
so they remain intact and silent — which is why only the `MNCommon`-backed
menus glitch.

## Fix (no submodule edits)

1. `tools/generate_yamls.py`: skip lines marked `/* STUBBED */` when
   parsing `reloc_data.h`. STUBBED entries are by construction
   placeholder `((intptr_t)0)` defines and must not displace real
   file_id assignments. After this change a from-ROM regen of the
   YAMLs reproduces the correct mapping automatically.
2. `yamls/us/reloc_animations.yml`: drop the spurious `FTYoshiAnimWin2`
   entry that was claiming file_id 0.
3. `yamls/us/reloc_menus.yml`: add `MNCommon` at file_id 0, with the
   ROM offset/size (`0x1B2C6C` / `0x57A0`) that was previously
   misattributed to `FTYoshiAnimWin2`. Verified against
   `tools/relocFileDescriptions.us.txt` (`-000: MNCommon`) and
   `tools/reloc_data_symbols.us.txt` (`llMNCommonFileID = 0x0`).
4. Regenerate `port/resource/RelocFileTable.cpp` via
   `python3 tools/generate_reloc_table.py` (no ROM required).

No changes to the `decomp/`, `libultraship/`, or `torch/` submodules.

## Audit hook

Any future tooling that reads `include/reloc_data.h` to mine `ll*FileID`
defines into a `file_id → name` map must filter out the `/* STUBBED */`
lines or it will repeat this collision the moment any new STUBBED FileID
is added. Same hazard applies to any hand-written diagnostic that greps
the header for FileID values.
